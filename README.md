# easy-vbc 实践过程

本文档记录了vibe-coding实践过程

## 实践 Task 1：基础前端三件套构建贪吃蛇

### 目标
使用 AI 辅助，快速构建一个功能完善的贪吃蛇小游戏。

### 技术栈
- **HTML5**: 构建游戏画布（Canvas）和基础页面结构。
- **CSS3**: 设计游戏界面样式。
- **JavaScript (ES6+)**: 实现游戏核心逻辑（蛇的移动、碰撞检测、食物生成等）。

### 过程描述
在此阶段，我利用 AI 生成了游戏的基础代码，并通过简单的 HTML/CSS/JS 文件组合实现了游戏在浏览器中的直接运行。这是一个轻量级的实现，重点在于理解游戏的基本循环和逻辑。

## 实践 Task 2：Vue 3 + Vite 重构与部署

### 目标
将原有的贪吃蛇游戏重构为现代化的 Vue 3 项目，并实现自动化线上部署。

### 技术栈
- **Vue 3**: 使用 Composition API 进行组件化开发。
- **Vite**: 作为新一代前端构建工具，提供极速的开发服务器和打包构建。
- **Vue Router**: 管理页面路由（如首页、游戏页）。

### 重构亮点
1.  **组件化架构**: 将游戏逻辑封装在 Vue 组件中（如 `snake.vue`），提高了代码的可维护性和复用性。
2.  **工程化管理**: 引入 `package.json` 管理依赖，使用 Vite 处理资源和构建。
3.  **状态管理**: 利用 Vue 3 的响应式系统更优雅地处理分数、游戏状态等数据。

### 部署实践：GitHub Pages
在此过程中，我学习了如何将 Vue 项目部署到 GitHub Pages：
- **构建**: 使用 `npm run build` 生成 `dist` 目录（静态资源）。
- **挂载**: 将 `dist` 文件夹中的内容挂载/推送到远程仓库的特定分支（通常是 `gh-pages` 分支）。
- **上线**: 通过 GitHub Pages 功能，将该分支作为静态网站对外发布，实现了项目的线上访问。

## 实践 Task 3：中医答题模块开发与全平台移动端适配

### 目标
增加中医知识研习模块，并对所有功能进行移动端 H5 适配，提升跨终端使用体验。

### 核心功能：中医答题 (TCM Study)
1.  **多模式研习**: 实现了“原文背诵（记忆卡片）”、“单选题”和“填空题”三种模式，满足不同深度的学习需求。
2.  **SRS 间隔复习算法**: 基于 Spaced Repetition 原理，根据用户答题结果（模糊、有印象、掌握）动态调整题目复现时间。
3.  **研习总结**: 在一轮练习结束后，自动生成详尽的总结报告，包括得分统计和错题复盘。

### 移动端 H5 适配实践
为了让项目在手机端也能流畅运行，进行了深度适配：
1.  **响应式布局**: 利用媒体查询（Media Queries）和 Vue 的响应式状态，针对不同屏幕尺寸自动调整 UI 布局。
2.  **贪吃蛇虚拟按键 (D-Pad)**: 
    - 针对移动端取消了键盘依赖。
    - 开发了 `VirtualJoystick.vue` 组件，实现了经典的“十字方向键”控制模式。
    - 使用 `touchstart` 事件优化触摸响应，确保零延迟的操作感。
3.  **交互优化**:
    - 针对中医答题模块，优化了移动端的按钮点击区域和输入框体验。
    - 引入了磨砂玻璃质感（Glassmorphism）设计，提升移动端视觉高级感。

## 总结：Vibe-Coding 的深度体会与工程化思考

通过这一系列实践，我深刻体会到了 **Vibe-Coding** 带来的开发范式变革，同时也对其在实际工程中的落地有了更理性的认识：

### 1. 范式变革：从“写代码”到“传达意图”
*   **意图驱动**：开发的核心不再是纠结于具体的语法细节，而是如何清晰地向 AI 表达业务逻辑和交互愿景。AI 极速响应，开发者则进化为纯粹的“架构师”与“决策者”。
*   **极速反馈循环**：极大地缩短了“想法 -> 代码 -> 运行”的周期，这种“即思即所得”的快感，让开发过程变得像是一场流畅的对话。
*   **沉浸式创作**：减少了查阅文档和处理琐碎 Bug 的干扰，让开发者能保持高强度的专注力，将精力集中在解决更有价值的问题上。

### 2. 现实挑战：Vibe-Coding 的“黑盒”局限
在实践过程中我也发现，纯粹的 Vibe-Coding 并非万能，它存在一定的局限性：
*   **调试的“黑盒”属性**：AI 生成的代码有时难以“一步成型”，当逻辑出现偏差时，定位问题和调试的过程往往具有黑盒特征，甚至会出现一些极其诡异、难以理解的 Bug。
*   **工程化落地的鸿沟**：全程盲目的 Vibe-Coding 确实能快速做出令人惊艳的 Demo，但若要打造真正稳定、可维护的**普通工程级别项目**，仍面临不小的挑战。

### 3. 工程化实践建议
为了填补 Demo 与成熟工程之间的鸿沟，我总结了以下几点前置操作建议：
*   **精细化提示词 (Detailed Prompting)**：需要通过极其详尽的描述来降低 AI 的理解偏差。
*   **引入 Skills 作为前置**：利用预设的 Skills 或 Context 注入，为 AI 提供更精准的上下文约束和操作指南。
*   **模块化分阶段进行**：不能期望 AI 一次性完成巨型任务，必须将功能拆解为细粒度的模块，循序渐进地引导生成。
*   **人工审核与把控**：开发者必须保持对代码的**个人审核把控力**，对 AI 生成的每一行关键逻辑进行校验，确保代码质量符合工程标准。

总的来说，Vibe-Coding 不仅仅是一种技术手段，更是一种**全新的创作心态**。它让编程回归到了“创造”本身，但真正的专业开发者需要学会如何在“自由的灵感”与“严谨的工程”之间寻找完美的平衡。
